# Copilot Bridge v1.3.0 设计

> 状态：Phase 28 已通过；v1.3.0 正式发布
> 当前阶段：Phase 28 — 协作模式预算
> 已发布基线：v1.2.2

## 1. 当前范围

v1.3.0 的首个纵切把协作轮次从代码约定改为用户可配置、运行时可执行的模式预算。设置入口位于“协作默认设置”下的第三个板块“协作模式设定”。

| 模式 | 默认预算 | 可配置范围 | 计数语义 |
|---|---:|---:|---|
| Assist | 2 | 1–20 | 同一咨询中的 Copilot 请求轮次 |
| Outsource | 6 | 1–20 | 同一咨询中的 Copilot 请求轮次 |
| Review | 1 | 1–20 | 双 reviewer 完整审查轮次 |

Review 每轮仍固定串行运行 `complexity` 与 `evidence` 两个隔离 reviewer；预算不改变 reviewer 数量、角色隔离或禁止并发写入的边界。

## 2. 配置和咨询快照

- 三个值保存在现有 `settings.json`，继续使用临时文件和原子替换。
- 三个值是新咨询的默认预算。创建咨询时把实际预算写入该咨询的本地状态记录。
- 修改默认预算不改变已经创建的咨询。
- v1.2.2 旧设置缺少字段时使用 2、6、1；旧咨询缺少预算时，在下一次合法续接时按对应模式当前默认值补齐。
- 预算必须是 1–20 的整数，GUI 与设置存储都执行校验。

## 3. 运行时门禁

- 每次发送前比较 `turnCount` 与咨询快照的 `turnBudget`。
- 达到上限时发送零条消息，MCP 返回 `blocked / turn_budget_exhausted`，且 `canRetrySafely=false`、`retryAction=none`。
- GUI 显示明确恢复说明：新建咨询，或调整仅对新咨询生效的默认预算。
- 预算门禁发生在 Edge/CDP 会话创建和页面写入之前。

## 4. MCP 状态

`copilot_bridge_status` 增加三个只读字段：

- `assistTurnBudget`
- `outsourceTurnBudget`
- `reviewTurnBudget`

现有工具数量、`consult_copilot` 输入 schema、项目权限和浏览器发送语义保持不变。

## 5. 明确不做

- 不恢复单一的全局沟通轮次上限。
- 不允许修改 Review reviewer 数量或角色。
- 不自动切换或升级 Assist、Outsource、Review。
- 不增加数据库、服务、端口、队列、MCP 写工具或浏览器自动化栈。
- Phase 28 的开发实现阶段不自行发布；正式发布必须在用户明确授权后单独执行完整门禁。

## 6. 验收

- 默认值、1–20 边界、原子持久化与旧设置兼容测试通过。
- 三种模式达到预算时均在调用执行器前阻止，发送次数为零。
- Assist 与 Outsource 的最后一个合法轮次可以完成。
- Review 继续输出两个隔离 reviewer，且一个完整双审计为一轮。
- Debug/Release 构建和完整测试通过；`.workbuddy/` 保持未跟踪、未打包。

## 7. 数据统计原型

数据统计作为 v1.3.0 的正式功能位于“概览”和“对话管理”之间；发布前须通过自动化、候选 GUI 启动与隔离升级门禁。

- 数据源只使用 Bridge 本地会话中由 Bridge 写入的 `agent` turn；显式导入的旧网页 `user` turn 不计入，但在导入会话上新产生的 Bridge 续接必须计入。
- 一次 Assist/Outsource 请求是一项任务和一次交付；Review 的一轮双 reviewer 是一项任务、两次交付。部分 Review 仍按所属会话计为一轮，不能跨会话合并取整。
- `reply_timeout` 是已确认提交但未完成的交付，必须保存为无 Copilot 回复的本地 turn，用于真实完成率；`not_submitted` 和无法确认是否提交的 `submission_unknown` 不进入交付完成率。
- Token 只估算 Bridge 可见请求与回复；默认倍率 3×，敏感区间至少覆盖 2×–5×，并始终包含用户选择的倍率。
- 原型费率为 Claude Opus 4.8、GPT-5.6 Think deeper 和 GPT-5.5 Instant 的输入/输出等效费率；未知模型不静默套价。

## 8. 发送与预算加固

- Composer 回读比较必须同时规范化预期文本和 DOM 回读，允许 Windows CRLF 与内容可编辑区 LF、尾部换行之间的等价差异，但仍保持发送前完整内容校验。
- GUI 与 MCP 必须在建立 Edge/CDP 会话前执行轮次预算门禁。
- 回复超时、提交状态不确定和部分 Review 都按发送后边界消费咨询轮次；GUI 与 MCP 均须保留内存状态，并尽力持久化本地咨询状态。
- 若本地会话正文比预算状态更新，后续调用必须停止并要求新建咨询，不能根据旧计数继续发送。

## 9. 架构与流畅性收口

- GUI 与 MCP 都只把请求交给同一个 `ConsultationCoordinator`；策略、跨进程租约、模式一致性、陈旧状态、绑定标签页和预算门禁只保留一份实现。
- Coordinator 在所有发送前门禁通过后才调用惰性的 page factory；行为测试直接断言被阻止时 page factory 调用次数为零，不再依赖宿主源码字符串顺序。
- `CollaborationRunner` 只负责编排模式；`CopilotTurnExecutor` 只负责单次网页 turn；`ConsultationPersistence` 只负责发送后的状态、设置和 Markdown 留存。GUI/MCP 不再构造运行上下文、预算记录或捕获发送后异常。
- 咨询状态内存账本用于磁盘失败后的同进程防重发。跨 Store 合并只写当前咨询，同一 ID 优先保留更高 `turnCount`，相同轮次再比较更新时间，防止 GUI/MCP 交错写、旧字符串状态或系统时钟变化导致预算回退。
- Dashboard 打开和刷新使用严格零写入的工作区读取路径；正文、Token、模型别名和 1× 基础费用只在数据集 `Prepare` 时计算一次，日期、语言和倍率重绘复用缓存。
- 统计模型匹配使用受控精确别名；未登记的未来模型保持“未定价”，不因名称相似静默套用旧价格。
- WPF 仍保持单窗口和无额外框架，但咨询与数据统计事件分别进入 feature partial；不在 v1.3.0 引入完整 MVVM、DI 容器、事件总线或通用 Repository。

## 10. 架构验收

- GUI 与 MCP 宿主中不再直接构造 `CollaborationRunner`、`CollaborationContext` 或 `ConsultationRecord`。
- 策略拒绝、busy、模式冲突、陈旧状态、缺少绑定和预算耗尽均在获取 Edge page 前返回。
- GUI/MCP 两个状态 Store 交错写不得回退其他咨询；旧字符串磁盘状态不得压过更高的内存已消费轮次。
- Dashboard 的只读打开不得创建目录、迁移旧项目或产生 `.bridge`；倍率滑块不得重新扫描会话 Markdown。
- 继续保持一个生产项目、一个测试项目、一个 EXE、两个直接生产依赖和四个 MCP 工具。
