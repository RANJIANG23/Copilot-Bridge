> 日期：2026-07-20（Asia/Shanghai）
> 状态：Phase 24 已通过；Phase 25 进行中
> 前置版本：v1.2.1 已发布
> 上位设计：[PROJECT-DESIGN.md](./PROJECT-DESIGN.md)
> 执行路线图：[EXECUTION-ROADMAP.md](./EXECUTION-ROADMAP.md)

## 1. 版本结论

v1.2.2 只完成两个由用户重新确认的 GLM-5.2 后续目标：

1. 为 GUI 增加可选的系统托盘生命周期，让用户能明确隐藏、恢复和退出 Bridge。
2. 将人可读的会话 Markdown 正文与 Bridge 内部结构化元数据分离，移除当前 Markdown 开头重复保存完整正文的 Base64 元数据。

本版本不改变 MCP 工具面、Edge CDP/DOM、项目访问权限、协作模式、模型选择、单写入锁或禁止自动重发语义。

## 2. GLM-5.2 意见的保留与优化

### 2.1 保留的用户价值

- 用户需要看见 GUI 是否仍在后台，并能从托盘恢复或明确退出。
- Markdown 应主要服务于人和本地工具阅读，内部元数据不应重复整份对话正文。
- 存储升级必须保留现有工作区、稳定会话 ID、项目分类、标题历史、模型验证状态和 Agent 访问权限。

### 2.2 不照搬的原实现假设

- 点号目录在 Windows 中不会天然隐藏；`.bridge` 目录由 Bridge 显式设置 Hidden 属性，设置失败不影响数据正确性。
- 按顺序写入 Markdown 和 JSON 不等于原子事务；实现必须使用临时文件、内容哈希、pending 记录和可恢复备份处理两文件中断。
- 托盘 GUI 不负责启动 MCP。MCP 仍由调用方通过 STDIO 启动，托盘也不增加 daemon、端口或进程间协议。
- 系统托盘不等于开机启动；v1.2.2 不写注册表、不创建启动项。
- sidecar 不重复保存 Markdown 正文；只保存会话、标题、项目和逐轮结构化元数据以及正文哈希。

## 3. 会话存储 v2

### 3.1 磁盘结构

```text
workspace\
  未分类对话\
    conversation-{stable-id}.md
  项目一\
    conversation-{stable-id}.md
  .bridge\
    conversations\
      {stable-id}.json
      {stable-id}.pending.json   # 仅事务中存在
    backups\
      v1-{timestamp}\
        manifest.json
        ...原相对路径 Markdown
```

- Markdown 保持一会话一文件和现有项目目录归类，只保存可读标题、角色、时间、模型信息和正文。
- JSON sidecar 保存稳定 ID、项目 ID、Copilot URL、双标题、本地标题、模式、时间、逐轮角色/模型/reviewer、Markdown 哈希和格式 revision；不保存 turn 正文。
- Markdown 使用少量不可见的稳定 turn 边界标记，确保正文中出现任意 Markdown 标题、代码块或表格时仍能无歧义恢复 turn 内容。
- `.bridge` 是 Bridge 管理目录；不得作为 Agent 默认内容范围或发布输入。

### 3.2 兼容与迁移

- v1.1 内嵌 Base64 Markdown 与 v2 分离格式可以并存并读取。
- 新会话直接使用 v2；对旧会话执行改名、移动或追加等显式写操作时，可以安全升级该会话。
- 批量迁移只能由用户在 GUI 中显式触发：先预览数量，再备份全部 v1 文件并写 manifest，然后迁移。
- 迁移失败必须恢复原文件并清除未完成 sidecar；不得留下一半 v1、一半损坏 v2 的同一会话。
- `search_conversations`、`read_conversation` 和其他只读路径不得创建 `.bridge`、备份、sidecar 或迁移文件。

### 3.3 双文件恢复

- 写入前生成 Markdown/JSON 临时文件和 pending 记录；pending 包含 revision、目标路径和两个内容哈希。
- 完成两个目标文件替换后才删除 pending。
- 读取发现 pending 时，只允许在明确写路径或 GUI 修复动作中完成恢复；只读 MCP 返回现有一致版本或稳定错误，不执行磁盘修复。
- JSON 与 Markdown 哈希不一致时不猜测正文；优先使用迁移备份或最后一致版本，并向 GUI 报告可理解错误。

### 3.4 明确不做

- 不采用标题作为文件身份，不在本版本引入可读文件名迁移。
- 不增加数据库、文件监视服务、后台索引器或通用迁移框架。
- 不允许 sidecar 成为隐藏的第二份正文副本。

## 4. 系统托盘

### 4.1 用户语义

- 设置页新增“系统托盘”，默认关闭，升级不改变现有关闭行为。
- 开启后托盘图标在 GUI 运行期间可见；点击窗口关闭会隐藏窗口，不触发 MCP 终止逻辑。
- 双击托盘图标或选择“打开 Copilot Bridge”恢复并激活原窗口。
- 托盘菜单“退出 Copilot Bridge”才真正关闭 GUI，并复用现有“后台常驻”与已登记同路径 MCP 的终止询问。
- Windows 会话结束时不得因为“关闭到托盘”取消系统退出。

### 4.2 技术边界

- 继续只有一个 `CopilotBridge.exe`；托盘属于现有 GUI 进程。
- 使用 Windows/.NET 自带托盘能力，不增加生产 NuGet 依赖。
- 不启动、重启、扫描或接管 MCP；只在显式退出时调用现有 `McpProcessRegistry` 逻辑。
- 不实现开机启动、托盘通知队列、自动更新或第二个 GUI 实例管理器。

## 6. 预算与停止条件

- 当前生产基线约 5861 行；v1.2.2 生产增量上限约 1100 行，保持总体约 7000 行目标。
- 直接生产 NuGet 依赖保持 2 个。
- 如果存储分离无法在无数据库条件下可靠恢复，停止 Phase 25，不以自动丢弃 sidecar、重建正文或后台全量重写绕过门禁。
- 如果托盘需要新的服务、端口、进程扫描或改变 STDIO MCP 所有权，停止 Phase 26。
- 任一改动导致未授权项目泄漏、只读 MCP 写盘、发送边界或前台无抢占回归时，停止在当前 Phase。
