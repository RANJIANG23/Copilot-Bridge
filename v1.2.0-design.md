# Copilot Bridge v1.2.0 设计候选

> 日期：2026-07-20（Asia/Shanghai）
> 状态：已从 v1.1.2 分离，尚未进入实施
> 前置版本：[v1.1.2-followup-design.md](./v1.1.2-followup-design.md)
> 上位设计：[PROJECT-DESIGN.md](./PROJECT-DESIGN.md)

## 1. 文档定位

本文档承接原 `v1.1.2-followup-design.md` 中不适合补丁版本的复杂需求。它是 v1.2.0 的设计候选和决策清单，不是允许一次性实现全部条目的总授权。

只有在 v1.1.2 完成并发布、用户确认 v1.2.0 优先级、对应设计门禁通过后，才能把候选项转化为执行 Phase。

## 2. 继续继承的硬边界

- 只有一个生产项目、一个测试项目和一个生产可执行文件 `CopilotBridge.exe`。
- 默认继续保持不超过 5 个直接生产 NuGet 依赖和约 7000 行生产代码预算；放宽预算必须单独获得用户确认。
- 日常 Copilot 交互仍只使用已绑定 Edge 标签页的 CDP/DOM。
- 发送状态不确定时绝不自动重发。
- 不增加数据库、本地 Web 服务、daemon、队列、通用 RPC、Provider 框架或第二浏览器栈。
- 沟通轮次默认无应用级上限。
- `.workbuddy/` 只用于开发期草稿输入，不进入源码、运行时配置、文档包或发布物。

## 3. v1.2.0 候选工作流

### 3.1 会话存储 v2

#### 用户需求

- Markdown 对人更易读。
- 内部元数据不重复正文。
- 文件管理器中的会话更容易识别。
- 存储升级不能损坏现有工作区。

#### 必须先回答的问题

- 单 Markdown 继续作为唯一事实来源，还是采用 Markdown 与 sidecar 元数据双文件？
- 双文件写入如何保证断电或崩溃后的一致性？只规定先写 JSON 再写 Markdown 不足以构成原子事务。
- sidecar 丢失时，Markdown 是否携带足够信息恢复 URL、标题历史、模型验证状态和逐轮元数据？
- 标题改名后文件名是否跟随变化？怎样避免外部链接失效、重名和非法字符？
- 如何对现有工作区执行幂等、可回滚、不中断的迁移？

#### 设计约束

- 在新方案通过迁移与故障恢复测试前，现有单 Markdown 格式继续作为事实来源。
- 如果采用可读文件名，必须保留稳定 ID，例如 `日期-标题短名-短ID.md`；不能只用标题作为身份。
- `.bridge` 点号目录在 Windows 中不会自动隐藏；如需隐藏必须明确处理 Hidden 属性，并提供用户可发现的管理入口。
- 迁移前创建可恢复快照或备份，不原地破坏唯一副本。
- 不引入数据库来解决文件一致性问题。

#### 门禁

- 旧格式、新格式和中途崩溃样本均能恢复。
- 重复运行迁移结果幂等。
- 重名、非法字符、超长路径和标题反复修改测试通过。
- 1.1.1/1.1.2 工作区可以无数据丢失升级。

### 3.2 系统托盘与启动生命周期

#### 用户需求

- 用户能看见 Bridge 是否仍在后台运行。
- 可以从托盘重新打开 GUI、查看状态或明确退出。
- 可以选择是否随 Windows 启动。

#### 必须先澄清的架构事实

- MCP 进程由调用方以 STDIO 启动；开机启动 GUI 不等于启动 MCP。
- 当前“后台常驻”只控制已经登记的 MCP 在 GUI 关闭时是否终止，不是 MCP 启动器。
- 托盘会改变窗口关闭、应用退出和单实例语义，必须独立设计，不能简单追加图标。

#### 门禁

- 明确 GUI、托盘实例和一个或多个已登记 MCP 的所有权关系。
- 开机启动关闭时零注册表/快捷方式残留；开启时能由 GUI 撤销。
- “关闭窗口”“退出 Bridge”“终止已登记 MCP”三种动作含义稳定且可恢复。
- 不批量扫描或终止同名进程。

### 3.3 UI 设计系统、可访问性与响应式

#### 目标

- 把现有窗口内颜色、卡片和按钮资源逐步整理为可维护的主题资源。
- 补齐高对比度、键盘操作、减少动效和窄窗口布局。
- 保持 Microsoft Copilot 风格，但不机械复制网页结构。

#### 事实修正

- 当前 XAML 已有颜色 Brush、卡片和按钮 Style，任务是整理和扩展，而不是从零建立 Token。
- 当前工程没有 `App.xaml/App.xaml.cs`；资源合并方案必须按实际启动结构设计，不能直接套用不存在的文件。
- “所有间距强制 8 点网格”不是门禁；清晰、对齐和可维护性优先。

#### 候选内容

- 将主题 Brush、排版、圆角、焦点和常用控件样式拆成受控 ResourceDictionary。
- 支持 Windows 高对比度，并为不希望动效的用户提供静态反馈。
- 为项目、模型和会话排序提供键盘“上移/下移”替代操作。
- 使用 GridSplitter 或受限响应式布局改善窄窗口，不重写为第二个 UI 框架。
- 增加首次使用说明、可恢复错误卡片和真正为空时的空状态。

#### 门禁

- 浅色、深色、高对比度和窄窗口均有实机证据。
- 键盘可以到达并执行全部主要操作，焦点可见。
- 减少动效模式下仍能理解拖拽目标与完成结果。
- 不因资源拆分显著扩大 XAML 重复和生产代码预算。

### 3.4 多调用方兼容与文案中性化

#### 目标

在保持 Codex 为首个正式支持调用方的前提下，评估 MCP 核心是否可以减少不必要的 Codex 专属文案和配置耦合。

#### 约束

- 不建立通用 Provider 框架。
- 不根据父进程名称猜测安全策略或产品能力。
- 任何调用方配置示例必须先用该调用方的官方格式和真实版本验证。
- 高级诊断页可以保留 MCP、STDIO 和 Edge CDP 等准确术语；普通页面只做不损失含义的简化。
- WorkBuddy 仍是开发期工具。不得加入 WorkBuddy 运行时适配、`.workbuddy` 配置分发或发布说明，除非用户以后明确改变这条规则。

### 3.5 Adaptive 模式研究

Adaptive 只能作为需要重新批准的研究项，原草稿的关键词规则和 2/6 轮硬预算不进入实现。

设计前必须满足：

- 不恢复应用级轮次上限；如需预算，必须是用户明确选择、可见且可取消的单次策略。
- 调用 Agent 能组织 `requestMarkdown`，因此关键词和长度不能被视为不可伪造的可信信号。
- 内部模式升级不能制造额外、不透明的 Copilot 发送。
- 发送状态不确定时仍然零重发。
- GUI 必须显示实际采用的模式、原因和可能产生的会话数量。

在这些条件形成可验证设计前，继续使用用户手动选择的 Assist、Outsource、Review。

### 3.6 对话与项目级协作能力

以下四项仍需要用户决定，不能因进入 v1.2.0 文档而自动实施：

1. 双标题同步：Copilot 初始标题、当前标题和标题历史如何与本地标题并存。
2. 会话级模型控制：允许控制新会话还是下一次请求，以及白名单和回退如何展示。
3. 项目级受控调用：Agent 可以读取概览、检索片段还是用户确认的完整范围。
4. 内容交接：复制到剪贴板、写入文件，还是通过新的 MCP 工具返回给调用方。

内容交接如果需要第三个 MCP 工具，必须单独更新工具面、安全注解、预授权和安装文档；不能伪装成 UI 小改动。

### 3.7 工作台增强候选

可在上述核心门禁之后按价值选择：

- 模板和会话标签。
- 明确的失败恢复：打开原 Copilot URL、复制未发送内容，但绝不自动重发。
- 多配置档案；必须先解决单标签页绑定和并发所有权。
- HTML/PDF 导出；不能为导出引入不成比例的依赖。
- 估算提示；必须明确它不是 Microsoft 或模型服务的真实计费数据。
- 多模型对比；必须保持隔离会话和用户可见的发送次数。

全局快捷键直接向 Copilot 发送不作为默认候选，因为它可能绕过当前调用方控制面并把 Bridge 变成第二个聊天客户端。可以先评估仅打开 GUI 或聚焦现有输入区的快捷键。

## 4. 建议的 v1.2.0 阶段门禁

| 阶段 | 内容 | 进入条件 | 退出条件 |
|---|---|---|---|
| 1.2-A | 用户决策与预算冻结 | 1.1.2 已发布 | 明确必做、候选、排除项和代码预算 |
| 1.2-B | 存储 v2 原型与迁移 | 1.2-A 通过 | 故障恢复与旧工作区迁移门禁通过 |
| 1.2-C | UI 资源与可访问性 | 1.2-B 通过 | 四种视觉/交互状态有实机证据 |
| 1.2-D | 托盘与生命周期 | 1.2-C 通过 | GUI/MCP 所有权和关闭语义通过 |
| 1.2-E | 调用方兼容 | 1.2-D 通过 | 仅实现已真实验证的调用方契约 |
| 1.2-F | 协作增强 | 用户逐项批准 | 对应 MCP、安全和数据门禁通过 |

阶段顺序可以在 1.2-A 调整，但一次只能有一个实施阶段处于进行中。预算不足时应删减候选项，不得用放宽架构约束掩盖范围过大。

## 5. 明确排除的原草稿方案

- 周期刷新时反复显示骨架屏。
- Adaptive 固定 2/6 轮硬上限。
- 仅凭请求文本关键词宣称路由信号可信。
- 未经验证就发布 WorkBuddy、OpenClaw 或 opencode 配置示例。
- 认为 `.bridge` 在 Windows 文件管理器中天然隐藏。
- 没有事务/恢复设计的 `.md + .meta.json` 双写。
- 只使用标题、会随改名变化的会话文件名。
- 全局热键绕过调用方直接发送 Copilot 请求。

## 6. 启动条件

v1.2.0 实施开始前必须同时满足：

- v1.1.2 已完成发布门禁并形成稳定回退点。
- 用户明确选择 v1.2.0 的必做候选和延期项。
- `PROJECT-DESIGN.md` 与新的 v1.2.0 执行路线图完成更新。
- 当前工作区没有把 `.workbuddy/`、本机数据或临时发布物纳入版本控制。
- 每个阶段都有独立验收、停止条件和预估代码增量。

本文档的创建只完成版本分割，不代表 v1.2.0 已经开工。
