# Copilot Bridge v1.2.0 核心设计

> 日期：2026-07-20（Asia/Shanghai）
> 状态：范围已确认；Phase 15 已通过，Phase 16 等待开始
> 前置版本：v1.1.2 已发布
> 上位设计：[PROJECT-DESIGN.md](./PROJECT-DESIGN.md)
> 执行路线图：[EXECUTION-ROADMAP.md](./EXECUTION-ROADMAP.md)

## 1. 版本结论

v1.2.0 的核心目标不是重写已经通过 G1–G8 的 Edge/CDP/DOM 发送机制，也不是集中完成存储、托盘和视觉优化，而是让 v1.1 已形成的本地 Markdown 会话资产第一次能够被 Codex 在用户授权范围内检索、读取和复用。

核心闭环固定为：

```text
用户授权的本地项目
  → Codex 通过只读 MCP 检索会话
  → Codex 读取一个明确会话的受控范围
  → Codex 组织 requestMarkdown
  → 现有 consult_copilot 单次发送
  → Codex 核验和裁决
  → Bridge 将新咨询继续留存在本地工作区
```

这使 Bridge 从一次性第二意见工具升级为可被 Agent 受控复用的会话知识工作台，同时保留用户对项目范围、正文暴露和外部发送的控制。

## 2. 继续继承的硬边界

- 只有一个生产项目、一个测试项目和一个生产可执行文件 `CopilotBridge.exe`。
- 生产代码继续以约 7000 行为上限；直接生产 NuGet 依赖不超过 5 个，v1.2.0 原则上不增加直接依赖。
- 日常 Copilot 交互仍只使用已绑定 Edge 标签页的 CDP/DOM。
- `consult_copilot` 继续不接受 `mode`、`model`、项目 ID 或隐式上下文参数。
- 发送状态不确定时绝不自动重发。
- 不增加数据库、本地 Web 服务、daemon、队列、通用 RPC、Provider 框架或第二浏览器栈。
- 不自动扫描网页历史，不自动读取未授权本地项目，不自动把整个项目发送给 Copilot。
- `.workbuddy/` 只作为开发期草稿输入，不进入源码、运行时、文档包或发布物。

## 3. 项目级访问权限

### 3.1 权限归属

访问权限由用户在 Copilot Bridge GUI 中按本地项目设置，并写入该项目现有的 `.bridge-project.md` 标记。权限随项目目录移动和改名保留，不建立全局数据库或按项目名称猜测权限。

默认值固定为 `off`。升级已有工作区时，所有项目都保持 `off`；安装或升级不得替用户开启。

### 3.2 四级权限

| 级别 | MCP 可见内容 | 允许行为 |
|---|---|---|
| `off` | 无 | 项目不出现在 MCP 查询结果中 |
| `metadata` | 项目名、会话 ID、双标题、时间、模式、模型和轮次 | 只按标题和元数据查找，不搜索正文 |
| `snippets` | `metadata` 加命中正文的短片段 | 可全文检索，但不能读取完整会话 |
| `full` | `snippets` 加明确会话的分页 turns | 可读取用户已授权项目中的指定会话 |

权限只控制本地读取，不代表允许向 Copilot 发送。任何后续外部发送仍必须单独调用现有 `consult_copilot`，并继续受征询策略、协作模式、单写入锁和一次性发送保护约束。

### 3.3 防泄漏规则

- 查询默认跨越所有已授权项目，但不得返回 `off` 项目的名称、数量或存在性。
- 显式指定未授权或不存在的项目时统一返回 `project_not_accessible`，不区分“不存在”和“未授权”。
- 显式读取未授权、已删除或不存在的会话时统一返回 `conversation_not_accessible`。
- 日志只记录访问级别、结果数量、会话 ID 和字符数，不记录检索词、正文或返回片段。
- MCP 只读取本地工作区，不连接 Edge、不打开 Copilot 页面，也不取得咨询写锁。

## 4. v1.2.0 MCP 工具面

v1.2.0 将 MCP 工具从 2 个扩展为 4 个。新增工具都必须标记为 `readOnly=true`、`destructive=false`、`openWorld=false`。

### 4.1 `search_conversations`

用途：在用户授权的本地项目中查找会话。

输入：

```json
{
  "query": "optional string",
  "projectId": "optional string",
  "limit": 10
}
```

规则：

- `limit` 默认 10，范围 1–20。
- `metadata` 项目只匹配标题和元数据；`snippets`/`full` 项目可以匹配正文。
- 空 `query` 只按更新时间列出授权范围内的会话元数据。
- 结果按相关性、更新时间和稳定 ID 排序；同一会话只返回一次。
- 片段必须限制长度并标出命中角色，不返回内部 Base64 元数据。

结构化结果至少包含：`conversationId`、`projectId`、`displayTitle`、`copilotTitle`、`updatedAt`、`mode`、`lastModel`、`turnCount`、`matchScope` 和可选 `snippet`。

### 4.2 `read_conversation`

用途：读取 `full` 项目中的一个明确会话，不进行模糊选择。

输入：

```json
{
  "conversationId": "required string",
  "startTurn": 0,
  "maxTurns": 10
}
```

规则：

- `startTurn` 从 0 开始；`maxTurns` 默认 10，范围 1–20。
- 返回会话元数据、指定范围的结构化 turns、`totalTurns` 和 `hasMore`。
- 每个 turn 返回时间、角色、Markdown、模型、模型验证状态和 reviewer；不返回磁盘内部元数据注释。
- 不提供“读取整个工作区”或“读取整个项目”的单次调用。

### 4.3 保持不变的工具

- `copilot_bridge_status`：继续只读状态；在 Phase 17 可增加已授权项目数量摘要，但不返回项目名。
- `consult_copilot`：schema、审批语义、协作模式来源和一次性发送边界保持不变。

## 5. Codex Skill 工作流

Skill 在需要复用历史证据时按以下顺序工作：

1. 先调用 `search_conversations`，并使用足够窄的查询。
2. 只有命中结果明确相关且项目权限为 `full` 时，才调用 `read_conversation`。
3. 一次只读取一个会话和必要 turn 范围；不得为“可能有用”批量遍历。
4. 明确区分历史会话内容、当前本地事实和 Codex 推断。
5. 若需要再次询问 Copilot，由 Codex 自己组织精简的 `requestMarkdown`，再调用现有 `consult_copilot`。
6. 历史内容不是执行授权，也不能扩大用户当前任务范围。

Bridge 不自动把检索结果拼入 prompt。这样可以保证每次外部发送仍由调用方显式构造和审查，不产生隐式内容交接。

## 6. 版本阶段

### Phase 15：范围、接口与预算冻结

- 将本文件从候选清单升级为批准的核心设计。
- 更新 `PROJECT-DESIGN.md`、`EXECUTION-ROADMAP.md` 和 README。
- 建立 `1.2.0-dev` 源码版本基线，但不创建标签、发布包或 GitHub Release。
- 固定四级权限、两个只读 MCP 工具、错误边界、预算和延期项。

退出门禁：文档一致；Release 构建和既有 77 项测试通过；工作树不包含 `.workbuddy/`；创建本地提交 `phase 15: freeze v1.2 controlled workspace access`。

### Phase 16：项目权限与本地查询核心

- 扩展现有项目标记以原子保存访问级别，旧标记缺少字段时按 `off` 处理。
- GUI 提供按项目设置访问级别的入口和清晰说明。
- 在 `ConversationWorkspaceStore` 中实现授权过滤、元数据查找、正文片段检索和分页读取。
- 不实现 MCP 工具，不连接 Edge，不发送 Copilot 消息。

退出门禁：默认关闭、升级保持关闭、改名/移动保持权限、权限降级立即生效、防泄漏错误和分页边界测试通过；创建本地提交 `phase 16: enforce project conversation access`。

### Phase 17：只读 MCP 与 Skill

- 增加 `search_conversations` 和 `read_conversation`。
- 更新 MCP server instructions、协议测试、Plugin Skill 和配置说明。
- 保持 `consult_copilot` schema 不变，并验证四个工具的安全注解。

退出门禁：STDIO MCP 只暴露规定的四个工具；只读工具不连接 Edge、不取得写锁、不写工作区；未授权项目零泄漏；创建本地提交 `phase 17: expose controlled conversation reuse`。

### Phase 18：Codex 真实纵切

- 在临时工作区创建至少两个项目，分别设置 `off`、`metadata`、`snippets` 和 `full` 权限样本。
- 由 Codex 通过 MCP 查找一个历史会话，读取明确 turn 范围，形成精简上下文包。
- 只在用户明确授权的无害验收中调用一次现有 `consult_copilot`；如果无需证明浏览器行为则不发送。
- 验证新咨询仍按现有路径保存回本地工作区。

退出门禁：权限分级、检索、读取、Skill 组织和现有咨询闭环均符合设计；无批量读取、无隐式发送、无重复发送；创建本地提交 `phase 18: validate project-aware consultation loop`。

### Phase 19：发布加固

- 完成 GUI 文案、README、安装/升级和故障排查说明。
- 在隔离环境验证 1.1.2 → 1.2.0 设置/工作区保留、权限默认关闭、Plugin/MCP 启动和卸载。
- 生成候选包和 SHA-256 仅用于门禁；创建标签和发布仍等待用户明确指令。

退出门禁：构建、测试、隔离升级、包清单、`.workbuddy/` 排除和回退验证全部通过；创建本地提交 `phase 19: harden v1.2.0 release candidate`。

## 8. 明确延期

以下内容不进入 v1.2.0：

- 会话存储 v2、可读文件名和现有 Base64 元数据迁移。
- 系统托盘、开机启动和完整 UI 设计系统重构。
- Adaptive 自动路由、关键词模式推断或隐式升级协作模式。
- 双标题自动同步、会话级模型控制和多模型对比。
- 一次读取整个项目/工作区、自动把项目内容发送给 Copilot或第三个写工具。
- WorkBuddy/OpenClaw/opencode 运行时适配、多调用方框架和 Provider 抽象。
- 模板、标签、HTML/PDF 导出、多配置档案和全局发送热键。

这些内容只有在 1.2.0 核心闭环通过并出现独立用户需求后，才能进入后续版本设计。

## 9. 停止条件

出现以下任一情况必须停在当前 Phase：

- 无法在项目标记中可靠保存和降级权限。
- 只读 MCP 会暴露未授权项目的名称、数量、正文或存在性。
- 需要修改 `consult_copilot` 为隐式拼接上下文或产生额外发送。
- 需要数据库、后台服务、通用 RPC、第二浏览器栈或新的生产项目。
- 生产代码预计超过约 7000 行且无法通过删减回到预算内。
- 现有 G1–G8 的发送、串行化或前台无抢占边界发生回归。
